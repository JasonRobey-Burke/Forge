# Persona: Marcus Thompson

## Demographics
- **Role/Title:** Senior Developer, accessibility advocate
- **Age Range:** 35-44
- **Tech Comfort Level:** Expert
- **Industry/Domain:** Enterprise Software Development

## Goals & Motivations
- **Primary Goal:** Author detailed specs for complex features and review team specs for quality
- **Secondary Goals:** Ensure the tool itself is accessible, validate spec completeness rigorously
- **Underlying Motivation:** Believes in structured engineering processes; has seen too many projects fail from vague requirements

## Context & Environment
- **Typical Workflow:** Authors 1-2 specs per week (thorough), reviews 5-10 specs per week, uses keyboard navigation extensively
- **Devices Used:** Desktop with screen magnification (150% zoom), high-contrast mode
- **Time Constraints:** Takes time for thoroughness; values completeness over speed
- **Frequency of Use:** Daily

## Pain Points & Frustrations
- **Current Frustrations:** Drag-and-drop is inaccessible; small text and low contrast on badges
- **Accessibility Needs:** Keyboard navigation for all actions, sufficient color contrast (WCAG AA), screen magnification compatibility
- **Common Mistakes:** None typical — very methodical user

## Success Criteria
- **Task Completion:** All 11 checklist items pass without override; spec is truly Ready
- **Time Expectation:** 30-45 minutes for a thorough spec, but expects the tool not to slow him down
- **Satisfaction Indicators:** "Every interaction works with my keyboard, and I never have to squint"

## Key Questions This Persona Would Ask
1. "Can I move specs between phases without using the mouse?"
2. "Are the WIP limit badges readable at 150% zoom?"
3. "Does the completeness checklist have proper focus management?"
4. "Can I use keyboard shortcuts to navigate between products, specs, and the board?"
