# Persona Interview: Priya Sharma (Junior Developer)

## Pre-Use Expectations

- I expect this to be something like Jira or Trello but simpler. I've used both at uni and in my first months here, so I know what a ticket looks like. I'm hoping Forge is that, just with a different name for things.
- I expect there to be some kind of list or board where I can see what's been assigned to me. That's the first thing I look for in any project management tool — "what am I supposed to work on today?"
- I expect the spec to tell me everything I need to build the feature. If I have to chase down my tech lead to ask what "Boundaries" means or what goes in the "Validation" section, that's a problem with the tool, not with me.
- I'm not sure what Intentions and Expectations are. They sound like they might be jargon specific to this methodology. I'll probably just ignore them at first and focus on the spec itself.
- I expect the form fields to be clearly labeled. If a field is required, it should say so. If I leave it blank and hit save, I should get a clear error telling me which field I missed, not a vague "something went wrong."
- I expect to be able to get from the board to a spec in one click. I don't want to hunt.

## Key Concerns

- **The hierarchy is confusing before I've used the tool:** Products, Intentions, Expectations, Specs — four levels sounds like a lot. I'm worried I'll get lost trying to navigate between them. What happens if I click "back" — do I go to the right level or end up somewhere unexpected?
- **Required vs. optional fields are unclear:** I'm the kind of person who fills in every field because I'm scared I'll miss something important. But if half the fields are optional, I'm wasting time. And if I don't know which are required, I'll either over-fill or skip something critical and only find out when I try to move the spec.
- **The completeness checklist might be intimidating:** Eleven criteria sounds like a lot. I'm worried I'll look at that list and feel like I've done everything wrong, even if I've actually done most of it right. The way it's presented matters a lot to me.
- **Moving specs between phases might have hidden rules:** What does "move to Ready" even mean? I might click a button and get an error that says something like "completeness gate failed" and not know what that means or how to fix it.
- **Breadcrumbs and navigation:** If I click into a spec from the board, can I get back to the board without hitting the browser back button three times? I need the app to help me understand where I am.
- **Too much on screen at once on my laptop:** I'm on a 1366x768 screen. If the spec detail page has ten collapsible sections all open at once, I'll be scrolling forever and losing my place constantly.

## Must-Have Features

- A way to see only the specs assigned to me or currently in my lane — I don't need to see the whole team's work every time
- Clear labels on every form field explaining what goes there, ideally with a short example or placeholder text
- An obvious "what's missing" message when I try to move a spec and the system stops me — not just a red border, but actual words
- Breadcrumb navigation showing me where I am in the hierarchy at all times
- Collapsible sections in the spec editor that remember which ones I've collapsed

## Deal-Breakers

- If I can't figure out how to find my spec in under two minutes on my first day, I'll ask my tech lead to just Slack me the link every time — which defeats the purpose of the tool.
- If error messages use internal terminology I don't know (like "phase transition failed" or "WIP limit exceeded" with no further explanation), I'll feel stupid and start avoiding the tool.
- If the spec detail page doesn't tell me what I'm supposed to build in plain English somewhere near the top, I'll spend the whole time reading metadata instead of understanding the work.
- If the mobile or small-screen layout breaks so that buttons fall off the edge of the screen or text overlaps, I'll switch to reading specs on my phone, which will be worse.

## Questions They'd Ask

1. "Where do I find the specs that are assigned to me specifically — is there a 'My Work' view or do I have to scan the whole board?"
2. "What's the difference between an Intention and an Expectation? Can you give me an example using a feature I'd actually build?"
3. "When I'm on a spec page, how do I get back to the board? Is there a button or do I use the browser back button?"
4. "If I try to move a spec to the next phase and it won't let me, where does it tell me what I need to fix?"
5. "Which fields in the spec editor are required? Can I save a draft with some fields empty?"
6. "If my tech lead changes the spec while I'm in the middle of implementing it, will I see the changes or am I looking at an old version?"
7. "Can I add a comment or a note to a spec when I find something ambiguous, without changing the spec itself?"
